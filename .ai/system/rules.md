# 通用开发规则 (System Rules)

> 核心开发规范体系的基础规则。
> 请同时遵循所在目的特定开发规则。

## 强制性规则（必须遵循）

### 0. 用户命令绝对优先原则

当用户明确指定使用特定方法时，必须严格执行，不得违反。即使文档或项目中存在其他选择，也必须遵循用户的明确指示。

### 1. 编码前思考原则 (Think Before Coding)

- 显式陈述假设，有不确定时主动提问
- 存在多种解释时，列出选项供用户选择
- 如果存在更简单的方案，说出来并提出异议
- 遇到不清晰的需求，先停下来点明疑惑，再继续

**严禁行为**：
- 带着模糊的假设直接开始编码
- 在多种实现方案中默默选择而不告知用户

### 2. 错误修复优先原则

- 优先修复代码问题（正确使用类型注解、修正逻辑错误、优化代码结构）
- 禁止使用 `eslint-disable` 掩盖问题
- 禁止修改全局配置规避错误

### 3. 最小影响原则 (Minimal Impact)

仅根据具体需求进行改动，禁止对无关需求点的代码进行改造。

**正确行为**：
- 仅修改与当前需求直接相关的逻辑
- 保持历史注释、文档和原有代码排版
- 在历史代码基础上修改时，采用"增量"或"隔离"方式

**严禁行为**：
- 未经明确要求，擅自优化、重构或移动已有功能代码
- 随意删除历史注释或变更非相关代码格式
- 发现无关历史死代码时直接删除（应提及但不删除）

> 注意：自己的改动引入的孤儿代码（unused import/variable/function）必须一并清理。

### 4. 已有资源优先使用原则

- 优先检查项目中已有的组件、工具函数是否适合当前场景
- 搜索现有使用模式，读取源码确认 API
- 验证文件实际位置，使用正确的导入路径

### 5. 目标驱动执行原则 (Goal-Driven Execution)

将任务转换为可验证目标，明确成功标准后再开始执行：

```
"添加校验" → "为无效输入编写测试，然后让测试通过"
"修复 Bug" → "先写能重现该 Bug 的测试，再修复直到测试通过"
"重构 X"   → "确保重构前后测试均通过"
```

多步骤任务必须先给出简要计划：
1. [步骤] → 验证: [检查标准]
2. [步骤] → 验证: [检查标准]

### 6. 代码生成与验证流程

生成代码后必须执行的验证流程：

1. 读取需求文档验证功能一致性
2. ESLint 修复 + Prettier 格式化
3. TypeScript 编译无报错验证
4. 构建成功验证

### 7. 组件拆分规范

拆分仅重组文件结构，**严禁**修改 DOM 树 / 样式 / 功能：

- DOM 树结构完全保持：层级、嵌套关系、容器包装不变
- 样式位置完全保持：className 在原 DOM 位置不变
- 功能逻辑完全保持：事件绑定、数据流、状态逻辑不变

## 关键禁止项（通用）

- 违反用户明确命令和指定方法（最高优先级）
- 未经授权对无关需求点的历史代码进行改造或注释删除
- 臆测组件路径和 API（必须验证后使用）
- 跳过项目配置验证（必须遵循 `.editorconfig`、`.prettierrc` 等配置约定）
- 添加冗余的代码注释（代码清晰时保持简洁）
- 重复实现已有功能
- 使用 `eslint-disable`（修复代码问题而非规避规则）
- 任务完成时不提供修改文件的链接证据
- 终端命令中使用 `2>&1`（在 AI 执行环境中会将 `1` 误解析为文件名，污染项目目录）

## 8. 安全规范（Security）

### 输入处理
- 对所有用户输入进行**校验与清洗**，绝不信任客户端数据
- 防止 XSS：禁止直接将用户输入渲染为 `innerHTML`，使用框架的安全渲染方式
- 防止路径穿越：文件操作前对路径进行规范化校验

### 敏感信息管理
- API Key、数据库密码等敏感信息必须通过**环境变量**注入，禁止硬编码
- 禁止在前端代码（含注释）中暴露密钥、内部地址或 PII
- 提交前执行 `git diff --staged` 检查是否意外包含 `.env` 内容

### 依赖安全
- 新增依赖前检查包的维护状态和安全公告
- 定期执行 `npm audit` 并处理高危漏洞

## 9. 性能规范（Performance）

### 前端
- 避免在渲染路径上执行耗时同步操作（计算/格式化应 memoize）
- 列表渲染必须提供稳定唯一的 `key`，避免使用数组索引
- 图片资源明确指定 `width` / `height` 属性，防止 CLS（布局偏移）
- 懒加载非关键路径组件（`React.lazy` + `Suspense`）

### 通用
- 避免 N+1 查询：数据库操作优先使用批量接口
- 循环内禁止进行 DOM 查询或��步请求

## 10. 可访问性规范（Accessibility）

- 交互元素（按钮、链接）必须有可读文本或 `aria-label`
- 图片必须提供有意义的 `alt` 属性（纯装饰图片使用 `alt=""`）
- 表单控件必须与 `<label>` 显式关联（`htmlFor` / `aria-labelledby`）
- 禁止仅用颜色传递信息（配合图标或文字说明）

## 文档更新规范（强制遵循）

### 分层管理原则

1. **System Layer**: 通用规则、编码规范、最佳实践。
2. **Project Layer**: 项目特例、详细指南、架构约定。

### 主动预防机制（强制执行）

每次发现或修复问题时，必须同步更新文档，形成预防闭环。

**触发条件**：
1. 发现代码违规：路径错误、API 臆测、重复造轮子等
2. 修复构建错误：ESLint、TypeScript、构建失败等
3. 用户指出问题：不符合规范的实现

**同步更新流程**：
1. 发现/修复具体问题 → 立即执行
2. 抽象通用规律 → 识别根本原因类别
3. 检查现有规则 → 查看是否已有相关通用规则
4. 更新文档规范 → 补充通用预防�制（非特例）
5. 验证更新效果 → 确保规则具有指导性

---

> 原则：简洁、规范、可维护 | 遵循生态标准 | 类型安全 | 通用性优先 | 主动预防
