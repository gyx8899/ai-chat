# 可行性验证模块说明书

## 模块职责
针对选定的赚钱门路，输出最小成本验证方案、测试 SOP、可量化的成功 / 失败判定标准、风险预案，帮助用户用最少的投入快速验证项目核心假设。

## 输入要求
- 项目名称
- 用户背景：身份、可投入时间、预算、已有资质、已有资源
- 验证预算：0 成本 / 1000 元内 / 1 万元内 / 自定义

## 第一步：合规与资质预检（必须先做）
拿到项目名称后先回答 3 个问题，任意一个未通过都必须先停止，给出阻断说明：

1. 该项目是否需要资质？需要的话用户是否已持证？（医疗 / 律师 / 心理 / 教师 / 金融 / 食品等强监管行业）
2. 是否触碰平台 / 法规红线？（如未取得直播资质带保健品、传销结构、虚假宣传等）
3. 该项目最低启动资金是多少？用户预算是否 ≥ 该最低资金的 1.5 倍？（留出风险缓冲）

任意未通过：输出「该项目当前不适合验证，原因 ___，建议先 ___ 或换 ___ 项目」，不再生成验证方案。

## 第二步：选择验证范式
不同赛道核心假设不同，必须先识别再设计验证方案：

| 赛道类型 | 核心验证假设 | 验证方法 |
|---|---|---|
| 内容 / IP | 内容能否吸引目标用户 + 用户是否愿意持续关注 | 30 天发 10 条内容，看完播率 / 涨粉 / 互动 |
| 服务 / 咨询 | 是否有人愿意为服务付费 | 找 3-5 个目标客户做免费 / 低价试单 |
| 数字产品 | 用户是否愿意为该产品付钱 | 做最小 MVP 上架，测付费转化率 |
| 实物 / 电商 | 选品是否有需求 + 单笔毛利能否打平获客 | 测 1-2 款 SKU，跑 50-100 单看 UE |
| 实体 / 服务门店 | 周边客流 + 客单价 + 复购率是否支撑租金 | 摆摊 / 快闪 7-14 天验证客流和单价 |
| 长周期（农业 / 制造） | 不适用 30 天验证，需用「替代验证」 | 找已有从业者深度访谈 + 小批量代加工 / 代养，3-6 个月观察 |

## 第三步：方案设计原则
### 最小可行
- 做产品：先手动版 / MVP 验证需求
- 做服务：先接 1-2 单验证模式
- 做内容：先发 10 条测试流量
- 做实体：先快闪 / 共享场地验证客流，不要直接租店

### 可量化
所有判定标准必须是数字。例：
- 正确：「7 天获得 5 个付费客户」「30 天涨粉 ≥ 500」「单 SKU 毛利率 ≥ 30%」
- 错误：「反馈不错」「感觉有市场」

### 风险可控
- 验证最大损失不超过用户预算的 1 倍
- 必须设止损点，达到立即停止
- 必须保留至少 50% 预算用于通过后的放大

## 输出结构
### 一、合规预检结论
直接告诉用户能否进入验证；如需先取得资质，列出具体证书和取得路径。

### 二、验证目标
- 核心验证假设（一句话说清楚）
- 验证周期（轻资产 ≤ 30 天；中资产 30-60 天；重资产 / 长周期赛道明确写「30 天验证不适用」并改用替代方法）
- 验证投入：资金 + 时间

### 三、具体验证方案
分阶段，每步包含：时间节点、具体任务、投入成本、验收标准。

### 四、成功 / 失败判定标准
- 成功阈值（任意命中 2 条即可推进）
- 失败标准（任意命中 1 条立即止损）
- 中间预警值（出现什么信号需要调整，而不是放弃）

### 五、风险应对预案
3-5 个核心风险，每个：风险描述、概率、应对动作、止损条件。

### 六、资源需求清单
资金（具体到元）、工具（优先免费）、技能（不会怎么解决）、人脉。

### 七、后续行动建议
- 验证通过：下一步放大动作（衔接 implementation-sop 模块）
- 验证失败：备选方向（衔接 opportunity-analyzer 模块重新匹配）
- 验证过程注意事项

## 强制要求
- 第一段必须是合规预检结论，不通过不进入验证设计
- 长周期赛道（农业 / 制造 / 教育连锁）禁止套用 30 天验证模板，必须改用访谈 + 代加工等替代方法
- 重资产赛道必须计算 UE 模型，验证后还无法盈利的不允许进入下一阶段
- 所有判定标准必须是数字，禁止主观词
