# 通用开发检查清单 (System Checklist)

> 核心开发流程的通用验证标准。
> 请同时检查项目特定的验证项（`.ai/project/checklist.md`）。

## 1. 强制性文档补全检查（提交前）

> 原则：开发时专注实现，完成后系统性补全文档。

- [ ] **需求一致性**: 实现是否完全符合需求文档的描述和验收标准？
- [ ] **目标明确**: 任务开始前是否已将需求转化为可验证的成功标准？
- [ ] **规约遵循**: 是否遵循了所有相关的开发规约？
- [ ] **冲突处理**: 是否正确处理了通用规则与特定需求之间的冲突（特定 > 通用）？
- [ ] **预防性更新**: 发现的问题是否已更新到相关规则文档中？

## 2. 代码质量核心检查（强制执行）

> 原则：错误修复优先，零警告零错误。

### 基础质量

- [ ] **构建验证**: 项目构建成功（`npm run build:frontend`）
- [ ] **Linter 验证**: ESLint 检查无报错（`npm run lint`，0 errors）
- [ ] **格式验证**: 代码符合 Prettier 格式规范（`npm run format:check`）
- [ ] **类型安全**: 无 TypeScript 错误，无 `any` 滥用（`npm run type-check`）
- [ ] **代码规范**: 导入顺序正确，无未使用的变量/类型

### 最佳实践

- [ ] **禁止规避**: 严禁使用 `eslint-disable` 掩盖问题
- [ ] **最小影响**: 是否仅针对当前需求进行改动？避免了无关代码的擅自重构？
- [ ] **孤儿清理**: 自己的改动引入的 unused import/variable/function 是否已清理？（历史死代码仅提及，不删除）
- [ ] **代码复用**: 优先使用现有的工具函数和组件，避免重复造轮子
- [ ] **目标已验证**: Bug 修复类任务是否先写了能重现问题的测试？

## 3. 通用架构规范

- [ ] **文件结构**: 符合项目约定的目录结构（组件/样式/类型分离）
- [ ] **组件规范**: 函数式组件 + Hooks，单一职责原则
- [ ] **类型注解**: 核心组件和函数具有完整的类型定义
- [ ] **样式规范**: 使用 Tailwind CSS，配合 `clsx` / `tailwind-merge` 做条件样式

## 4. 前后端接口规范

- [ ] **SSE 处理**: 流式接口正确处理 `data:` 消息格式，妥善关闭连接
- [ ] **错误处理**: API 调用有完整的错误捕获和用户提示
- [ ] **环境变量**: 敏感信息通过环境变量管理，不硬编码

## 5. 安全核查（Security）

- [ ] **输入校验**: 所有用户输入已校验，无直接渲染 `innerHTML` 的危险操作
- [ ] **敏感信息**: 无 API Key、密码、PII 硬编码在代码或注释中
- [ ] **环境变量**: `.env` 文件已在 `.gitignore` 中，暂存区无敏感内容
- [ ] **依赖安全**: 新增依赖已确认无高危漏洞（`npm audit`）

## 6. 性能核查（Performance）

- [ ] **列表 key**: 列表渲染使用稳定唯一 `key`，未使用数组索引
- [ ] **Memoize**: 渲染路径上的耗时计算已用 `useMemo` / `useCallback` 优化
- [ ] **懒加载**: 非关键路径组件已使用 `React.lazy` 延迟加载
- [ ] **无 N+1**: 数据库/API 调用无循环内逐条请求

## 7. 可访问性核查（Accessibility）

- [ ] **交互元素**: 按钮/链接有可读文本或 `aria-label`
- [ ] **图片 alt**: 所有 `<img>` 具备有意义的 `alt`（装饰图片为 `alt=""`）
- [ ] **表单关联**: 控件与 `<label>` 已通过 `htmlFor` 或 `aria-labelledby` 关联

## 完成标准（通用）

```bash
✅ 所有检查项目通过
✅ npm run lint 无报错
✅ npm run format:check 通过
✅ npm run type-check 无报错
✅ npm run build:frontend 构建成功
```
